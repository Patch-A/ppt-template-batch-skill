# Buyer Board Safety And Data Quality Design

日期：2026-07-18

## 目标

修复最近代码审查发现的买家看板数据真实性、资产抓取安全和缓存正确性问题，并增加可重复的回归测试。保持现有 CLI、控制台接口、Feishu/Aily skill 包和 PPT 模板版式兼容。

## 范围

### 资产抓取安全

- 只允许 `http` 和 `https` 资源。
- 对用户或模型提供的 URL 做基础规范化，并阻止本机、回环、链路本地和私有网络地址。
- 对跟随重定向后的最终地址重新执行同样的安全校验；默认不接受跨主机重定向到非官方资源。
- Python、curl、Data URI 三条下载路径都执行响应大小上限，超限时在内存和磁盘写入前终止。
- 保留现有每买家和全局时间预算。

### 资产缓存与报告

- Logo 和站点图分别判断缓存是否有效；只命中其中一项时，另一项仍然重试。
- 缓存保存成功结果和可重试的失败状态，失败不阻塞下一次运行。
- 正常、跳过和失败记录使用同一组报告字段，便于控制台和后续工具消费。

### 买家资料真实性

- 简介长度补齐只使用中性企业描述，不添加未经证实的采购能力、采购计划或合作意愿。
- 产品字段始终根据买家证据收敛，即使全局采购需求只有 1-3 项也不能无条件复制给所有买家。
- 保留现有“需核实具体设备”降级语义；证据不足时不得生成确定性采购结论。

### Logo 识别

- 品牌匹配只使用候选文件名、alt 和 class 等候选自身字段，不使用页面 URL 作为拒绝依据。
- 将 `rgb`、`color`、`mark`、`symbol`、年份等描述性后缀视为中性词。
- 只有没有品牌交集，或命中明确的竞争品牌/子品牌规则时才拒绝候选。

## 不在本轮范围

- 控制台任务队列持久化和远程认证。
- PPT 渲染服务替换或全面视觉回归基础设施。
- API provider 路由重构和模型升级。

## 数据流

1. 买家研究输出经过产品、简介和来源字段规范化。
2. 资产抓取先规范化 URL，再执行安全校验、下载上限和候选过滤。
3. 资产结果按 Logo/站点图独立写入缓存和买家记录。
4. 报告记录统一的命中、来源、拒绝原因、超时和大小限制状态。
5. 回归测试覆盖纯函数、缓存短路、资源下载边界和最小 PPTX 产物。

## 错误处理

- 安全校验失败返回可识别的拒绝原因，不回退到不受限的下载路径。
- 超过响应大小限制时释放临时文件并记录 `response_too_large`。
- 单个买家资产失败不影响其他买家；全局预算耗尽的记录仍写入完整报告 schema。
- 证据不足继续保留买家，但标记人工核实，不填充未经证实的采购事实。

## 测试策略

- 使用标准库 `unittest`，避免新增测试依赖。
- 先写失败测试，再逐项实现：Logo 中性后缀、页面路径误拒绝、缓存缺失资产重试、短采购需求收敛、简介中性补齐、资源大小限制和统一报告字段。
- 运行全量 Python 语法编译、测试套件、CLI `--help` 冒烟和最小 PPTX 生成检查。

## 兼容性与迁移

- 保持现有 JSON 字段名和 CLI 参数；仅新增报告字段或内部状态。
- 旧缓存缺少新字段时按缓存未命中或空状态处理，不删除用户已有资产。
- 默认资产模式和浏览器回退开关保持不变。
